# To audit and implement a RootTrax admin portal (canonical architecture guide)

This is the master implementation guide for building and debugging a **RootTrax-style admin portal** (Creator / Metabase / FairPath pattern).

## What this architecture is
A RootTrax admin portal is a layered UI system where:
- ASP.NET host + middleware bootstraps auth/session/routing/static assets,
- page resolution is largely **PageLayouts-driven**,
- route handlers render **kScript .ks.html** page classes,
- UI sections (sidebar, title, content, scripts) are generated by kScript classes,
- client behavior is standardized through **KCs JavaScript** stack + Bootstrap template.

## Canonical runtime pattern (host startup)
Use the same startup shape across portals:

1. Build config + set `AppSettings`.
2. Set DB connection strings.
3. Configure RooTraxState / kScript root.
4. Register portal services via `AddBuffalyPortalServices(...)`.
5. Configure user state.
6. Apply `UseBuffalyPortalDefaults(config, siteId, isAuthRequired)`.
7. Let PageLayouts drive route-to-handler behavior.

## Canonical web root contract
A valid RootTrax portal should include (minimum):
- `wwwroot/css` (bootstrap, app, icons)
- `wwwroot/js` (head + plugins)
- `wwwroot/js/kcs` (global.js, JsonMethod.js, grid4.js, MasterPage.js, etc.)
- `wwwroot/JsonWs` or `wwwroot/JsonWS` (generated JS service facades)
- `wwwroot/kScripts` (page classes, object admin ks files, master page ks snippets)
- `wwwroot/lib`, `fonts`, `images`, optional `custom_css`

- Bootstrap-based shell (`bootstrap.min.css`, app.css, icons)
- KCs stack loaded globally in layout head
- Master page JS loaded (`MasterPage.js`)
- standard sections: `SidebarMenu`, `MainPanelTitle`, `MainPanelContent`, `BottomScripts`

## Request-to-page resolution flow
Canonical intent:
1. Incoming URL resolved against `PageLayouts` rows (`Url`, `Handler`, `SiteID`, enabled).
2. Handler points to kScript endpoint pattern with `Output` and `Class`.
3. Page model loads class + output `.ks.html` files.
4. kScript evaluates sections (title, breadcrumbs, content, scripts).
5. Razor layout wraps generated sections in standard shell.

In practice, many portals use `k.aspx` style handlers in PageLayouts for legacy compatibility.

## kScript page class model
The page class model is rooted in common base behavior:
- `KScriptPageModel<TUserState>` orchestrates:
  - request parameter extraction (`Output`, `Class`, `Handler`),
  - symbol injection (`Request`, `UserState`, `AppSettings`),
  - include/evaluate of class + output,
  - section materialization (`MainPanelContent`, etc.),
  - standardized exception handling.

## KCs JavaScript responsibilities
KCs stack typically provides:
- JSONWS invocation contract (`JsonMethod.js`),
- form and grid primitives (`grid4.js`, validators, selector helpers),
- page-level utility + state helpers,
- shared master-page interactivity (`MasterPage.js`).

These scripts are expected globally by kScript-generated UI fragments.

## Why portals appear “unstyled” or “fragment-only”
Common root causes:
1. `_SimpleLayout` not used or not reachable.
2. Static asset roots missing (`css`, `js/kcs`, icons/fonts).
3. `MasterPage.js` / KCs bundle not loaded.
4. `UseBuffalyPortalDefaults(...)` omitted/misordered.
5. `RooTraxState` not configured to correct kScript root.
6. PageLayouts missing / wrong `SiteID` / disabled rows.
7. Handler format mismatch (`/k` vs `/k.aspx` expectation).
8. Wrong DB connection (points to DB without required stored procs/rows).

## Canonical build checklist for a new RootTrax portal
1. Host bootstraps app settings, connection strings, RooTraxState.
2. `AddBuffalyPortalServices(...)` and `UseBuffalyPortalDefaults(...)` wired.
3. `_SimpleLayout` copied and verified.
4. full static asset surface copied (css/js/kcs/fonts/images/JsonWs/kScripts/lib).
5. auth exclusions + login/logout/error routes configured.
6. page layouts seeded for target SiteID.
7. representative routes verified (`/`, `/k...`, object grid route, login, error).
8. JS console clean (no missing KCs script errors).

## Audit workflow (required when onboarding/fixing a portal)
When asked to audit a RootTrax portal:
1. Compare Program.cs against FairPath/Creator/Metabase baseline.
2. Compare `_SimpleLayout.cshtml` and loaded scripts against canonical list.
3. Validate `wwwroot` folder contract.
4. Verify DB PageLayouts rows and handler format for target SiteID.
5. Verify kScript root + include chain (`application.ks`, `references-generated.ks`, object ks files).
6. Verify base models in use (LoginBaseModel, ErrorModel, KScriptPageModel).
7. Produce “delta to canonical” report with exact file-level fixes.

## Implementation gotchas discovered during live migration
When moving a non-canonical portal to RootTrax architecture, these are required to avoid false 404/500 outcomes:

1. `RooTraxStateSettings` must exist in `appsettings.json`.
   - Missing this causes startup failure when calling `StartupConfigurator.ConfigureRooTraxState(...)`.
2. Shared layout partial dependencies must be present.
   - `_SimpleLayout.cshtml` expects `Pages/Shared/_Footer.cshtml` (and commonly `_ValidationScriptsPartial.cshtml`).
3. Razor page endpoints for kScript rendering must exist.
   - `Pages/k.cshtml` + `Pages/k.cshtml.cs` (using `KScriptPageModel<TUserState>`) are part of the canonical route path.
4. PageLayouts handler format must match active endpoint mode.
   - Align stored `Handler` values with runtime expectation (e.g., `/k?...` vs `/k.aspx?...`).
5. Temporary manual HTML routes should be removed after canonical `k` flow is restored.

These checks should be treated as mandatory in first-pass audits.

## Mandatory diagnostics and JsonWs stub generation guidance
When implementing or repairing a RootTrax-style admin portal, include both requirements below in the plan and verification pass.

1. Program.cs must be wired for observable diagnostics/logging.
   - Ensure standard ASP.NET logging providers are configured for development troubleshooting.
   - Ensure unhandled exceptions and request pipeline errors are surfaced in logs (not swallowed silently).
   - Ensure there is a deterministic local place to review runtime/request failures while validating grids/JsonWs calls.
   - If "no logs" are available during troubleshooting, treat Program.cs logging configuration as incomplete until fixed.

2. JsonWs JavaScript stubs must be generated for admin services.
   - Grid pages depend on generated `/JsonWs/*.ashx.js` client stubs.
   - Create and maintain an `add-js-stubs` batch workflow for the portal and run it after adding/updating JsonWs services.
   - Verify required stubs load with HTTP 200 (not 404), especially object/admin endpoints used by Sessions/Messages grids.
   - If grids render shell but no rows, validate stub generation before deeper UI debugging.
